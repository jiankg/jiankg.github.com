git-reflog找回误删除的commit

发表于 2015-04-07   |   分类于 技术   |  

问题是这样出现的: 游戏正式上线了，以为develop分支上暂时不会有什么修改，但碰上了策划要求活动配置需要按服务器差异各自开放，所以没有拉新的分支，直接在develop上做活动的修改。
当活动内容做了80%，提交了A,B,C 3次commit的时候，线上的代码出现了bug需要紧急修复。当时能想到的解决方法就是把develop通过 git reset --hard COMMIT 回退到A commit之前的版本，然后拉一个新分支，切换过去之后再切换到最新的C commit上。但是 git co -b new_branch 之后，git log打出来的日志里没有A,B,C的log了。活动的3次版本提交都丢了，此次的活动修改涉及到的文件比较多，当时有点慌神，如果找不回来那么就都得重写一遍。
经同事提醒，网上查了一下之后，知道 git reflog + git cherry-pick 命令配合使用可以把之前误删除的commit找回。

log里有ABC三次提交是新活动的，first ci是正式的。先切换到新分支，接下来使用 git cherry-pick 来找回ABC三个commit,把修改好的内容扔进新分支里。

这样new_act分支上就已经是最新的活动代码了。
有一点需要注意的是，git cherry-pick 的时候，必须按照先后顺序找回A,然后B,最后才是C,要不然中间可能会出现一些冲突，如果文件多解决起来就比较麻烦，按照顺序找回版本则可以避免这个问题。

回过头来再看当线上出现bug，需要修复，最合理的办法应该是打 hotfix 分支，但我们一直没有用hotfix那一套，所有问题就在develop上修改，改完了打tag然后更新线上代码。这样做会导致tag版本升的飞快，但是hotfix没用起来也只能继续让它飞升了。
再说说改完活动切换分支的做法应该是当需要修改线上bug需要用到develop分支而develop有你新改活动代码的时候，如何正确把活动代码扔进新分支，把develop回退到线上状态？

1.提交当前的活动。
2.创建并切换到新分支。
3.切换回develop分支。
4.git reset —hard (线上代码同步commit版本号)

如此简单的做法，当是没想到，实在蛋疼。

以上仅是个人使用经验，不一定代表正确的思路，仅供参考。

解决Octopress分类目录支持中文的问题

发表于 2014-10-16   |   分类于 技术   |  

几个月前，装了一次Octopress，可是在进行到 bundle install 的时候老是出现问题，后来放弃了。
拖了这么久，又装了一次，这一次出奇的顺利，所有基础件都搞定了的时候开始摸索各种组件。
在弄到分类问题的时候，问题来了，不能支持中文。
因为生成的 /_deploy/blog/categories/ 下的所有有关分类的目录名全是英文，
